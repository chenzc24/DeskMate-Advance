# DeskMate Advance：模型任务梳理与宏观计划

## 1. 计划结论

Advance 项目的模型工作应围绕一条可交付主线展开：

> 摄像头/麦克风输入 → 预训练感知与特征提取 → 时序学习模型 → 稳定化与置信度门控 → 统一语义事件 → 机器人控制层

模型侧的首要交付不是一次性训练所有可选模型，而是完成一个可复现、可部署、可对接的时序学习闭环。根据 PPT，推荐把 **动态手势识别（TCN 或 GRU）** 作为唯一必做的自训练模型：输入为归一化的 `T × 21 × 3` 手部关键点序列，最低覆盖 `wave`、`swipe`、`circle` 和 `no_gesture`。

其他模型只在预训练方案或规则方案经过实测仍不稳定时启动：

- 连续姿态分类：条件启动；
- 手机检测器微调：条件启动；
- 多模态 off-task 行为分类：时间充足时启动；
- 音频类别识别与个性化校准：演示闭环完成后再考虑。

## 2. 与已完成 baseline 的边界

`E:\NOS_NUS\DeepLearning\project` 与本项目属于不同任务。本计划不继承其中的任务定义、数据、标签、权重、代码包、配置或接口。

Advance 项目应在当前目录建立独立工程，并满足以下约束：

- 不导入 baseline 的 Python 包；
- 不复用 baseline 的模型权重、类别映射和数据划分；
- 不假设 baseline 的输入输出能够被机器人控制端消费；
- 所有训练、评估、部署和对接契约均由 DeskMate Advance 的 PPT 需求重新定义；
- baseline 最多只能证明既有的通用训练工程能力，不能作为本项目的技术或接口依据。

## 3. 模型侧职责与非职责

### 模型侧负责

- 需求场景、模型事件和验收指标的冻结；
- 数据采集规范、标注规范、划分和质量审计；
- MediaPipe 等预训练感知组件的输入预处理和特征提取；
- 动态手势时序模型的训练、评估、导出和在线推理；
- 低置信度、缺失关键点、抖动、持续时间和冷却逻辑；
- 统一事件输出、日志、回放和事件模拟器；
- 在 laptop / Raspberry Pi 5 目标环境进行性能验证；
- 与控制组完成事件级联调和端到端场景验收。

### 模型侧不负责

- 电机、舵机、显示屏和 Arduino 的底层驱动；
- 将某个模型类别硬编码成具体电机速度或动作序列；
- 机械臂、两轮底盘等可选机械扩展；
- 以训练大规模原始视频模型替代 10 天内可落地的关键点时序模型；
- 在核心演示尚未跑通时同时开发全部可选模型。

模型只表达“发生了什么”和“置信度有多高”；控制层负责决定“机器人具体怎么动”。

## 4. 任务优先级

### Must：必须完成

1. 冻结动态手势的类别语义和机器人使用场景；
2. 建立独立的数据采集、关键点提取和序列化流程；
3. 训练并选择一个小型 TCN；GRU 仅作为时间允许时的对照模型；
4. 输出 `gesture_detected` 等统一语义事件；
5. 实现置信度阈值、连续帧确认、进入/退出阈值和冷却；
6. 完成离线评估、目标设备性能测试和端到端联调；
7. 提供可回放输入与可模拟事件，保证硬件联调不依赖实时识别成功。

### Should：满足触发条件后完成

1. 连续姿态模型：只有在角度与持续时间规则无法稳定区分目标姿态时启动；
2. 手机检测微调：只有在目标摄像头上的小目标、遮挡或角度测试明显失败时启动；
3. 简单的多模态 off-task 分类：只有动态手势主线已经冻结且仍有完整数据与联调时间时启动。

### Could：核心演示完成后的扩展

- YAMNet 音频类别识别；
- 用户级距离、姿态和使用习惯校准；
- 用户身份识别；
- 更细的手势方向或连续动作类别；
- 原始视频端到端时序模型。

## 6. 数据与训练方案

### 6.1 动态手势主模型

建议先冻结以下最低标签：

- `no_gesture`：静止手、自然移动、拿取物品、离开画面等真实负样本；
- `wave`；
- `swipe`；
- `circle`。

若控制语义需要区分方向，可在数据量足够后再拆成 `swipe_left` / `swipe_right`，不要一开始扩大类别数。

建议采集目标：

- 5–8 名参与者；
- 每人、每个动作 20–30 条有效序列；
- 每个动作覆盖左右手、不同速度、距离、角度与背景；
- 负样本数量至少与所有正样本总量同量级；
- 额外保留连续长视频，专门评估误触发率。

划分必须按参与者或完整会话分组，不能把同一视频切出的相邻窗口同时放入训练集和测试集。

### 6.2 输入与预处理

核心输入为 `T × 21 × 3` 的手部关键点序列，并额外保留：

- 时间戳与实际帧间隔；
- 左右手标记；
- 关键点有效性/缺失掩码；
- 检测置信度；
- 可选的速度、方向和轨迹派生特征。

归一化建议以手腕为原点、手掌尺度为缩放基准；训练时加入时间拉伸、轻微坐标噪声、随机丢帧和镜像增强。原始视频只用于审计与回放，不作为第一版模型的直接输入。

### 6.3 模型选择

第一选择为小型 1D-TCN：结构简单、并行推理友好、适合固定窗口且易于部署。GRU 只在以下情况之一发生时加入比较：

- TCN 对速度变化明显不稳；
- 需要更长上下文；
- TCN 在跨用户测试上明显不足；
- 团队仍有至少半天完整训练和联调余量。

最终选择以跨用户表现、误触发率、延迟和部署成本共同决定，不只看单次 accuracy。

## 7. 条件扩展的启动门槛

### 连续姿态分类

仅当实际测试证明“角度 + 时长 + 平滑”不能稳定产生 `bad_posture` 或 `static_too_long` 时启动。输入使用归一化身体关键点序列，候选模型为小型 MLP、1D-TCN 或 GRU。若无法快速获得跨用户、跨桌椅高度的可靠标签，应继续使用可解释规则并进行个体校准，而不是仓促训练。

### 手机检测微调

先在真实机器人摄像头距离、角度、遮挡和光照下评估预训练检测器。只有当目标场景召回明显不足，且能获得足够标注框时才微调轻量检测器。否则应优先调整输入分辨率、ROI、持续时间门控与摄像头位置。

### Off-task 多模态分类

候选输入包括手机置信度、头部方向、静止时长、计时器状态和历史事件。由于 “off-task” 标签主观、容易泄漏场景偏差，第一版只做规则融合或简单 MLP；不把它设为核心演示依赖。

### 音频分类与个性化

音频分类需先确认外接麦克风、采样质量和明确类别；否则只保留音量/SPL 统计。个性化优先做阈值校准，不训练用户专属大模型。

## 8. 对接契约

模型运行时至少输出以下逻辑字段：

```json
{
  "event": "gesture_detected",
  "model_level": "advanced",
  "confidence": 0.91,
  "duration_s": 0.8,
  "evidence": {
    "gesture": "wave",
    "window_frames": 24
  },
  "suggested_action": "acknowledge"
}
```

约束如下：

- `event` 表示语义，不包含电机指令；
- `confidence` 必须可校准，不能把 softmax 最大值直接等同于可靠性；
- `duration_s` 来自时间戳，不由固定帧率假设计算；
- `evidence` 记录可复现判断所需的最小证据；
- 低置信度、关键点缺失和类别冲突统一返回 `unknown` 或不发事件；
- `suggested_action` 只是建议，控制层保留安全规则与最终决定权；
- 事件层和 USB serial/Arduino 协议之间使用独立适配器。

需要同时提供两种解耦工具：

1. **模型回放器**：读取录制视频或关键点序列并输出事件，不依赖机器人；
2. **事件模拟器**：按脚本发送合法事件，不依赖模型，供控制组联调。

## 9. 建议验收线

以下是建议的工程验收线，不是 PPT 原文，可在第一次试采后调整：

- 跨用户测试集 macro-F1 ≥ 0.85；
- 每个核心手势 recall ≥ 0.80；
- 连续 5 分钟真实负样本中误触发不超过 1 次；
- 从动作完成到稳定事件发出的延迟 ≤ 1 秒；
- 模型自身 CPU 推理延迟 ≤ 20 ms/窗口；
- 包含关键点提取的完整感知循环在目标设备上 ≥ 10 FPS；
- 关键点短时丢失、手离开画面或置信度不足时不产生危险事件；
- 训练、评估和部署使用同一标签映射与归一化配置；
- 三个核心端到端场景重复执行 5 次，至少 4 次得到正确语义事件。

最终报告必须同时给出混淆矩阵、每类 precision/recall/F1、跨用户结果、长视频误触发率、延迟和失败案例。

## 11. 决策门

### Gate A：7/20 试采后

- 关键点是否稳定？
- 手势语义是否能被不同人一致执行？
- 机器人真正需要的是类别事件还是方向/速度参数？

若语义不一致，先缩减类别，不进入大规模采集。

### Gate B：7/23 候选模型冻结

- 跨用户结果是否达到可接受水平？
- 长视频误触发是否可通过门控解决？
- TCN 是否已经足够，是否真的需要 GRU？
- 是否存在必须通过补采解决而非继续调参的问题？

### Gate C：7/25 集成冻结

- 事件字段和控制端是否一致？
- 模型、事件层和 Arduino 控制是否解耦？
- 模拟事件和录制回放能否独立复现问题？
- 目标设备性能是否满足实时演示？

Gate C 之后不再改类别、网络结构和接口，只允许修复阻塞演示的问题。

## 13. 一句话执行原则

先用一个小型时序模型把“数据—训练—实时推理—统一事件—机器人响应”跑成可重复闭环，再按真实失败证据决定是否增加第二个模型。
